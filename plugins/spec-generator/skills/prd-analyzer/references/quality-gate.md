# 质量门禁：PRD 分析

## 产出物

`{workspace}/prd-analysis.md`

## 目标

将非结构化的 PRD 转化为结构化需求，消除歧义，建立后续步骤所需的统一事实来源。

## 使用场景

| 消费者 | 消费什么 |
|--------|----------|
| Step 3 (代码库映射) | `{workspace}/prd-analysis.md` 中的功能关键词和实体名称，用于代码库搜索定位 |
| Step 4 (生成规格) | `{workspace}/prd-analysis.md` 中的功能列表、数据模型、接口定义、约束条件，作为 spec 的输入事实 |
| Step 5 (质量审查) | `{workspace}/prd-analysis.md` 中的功能列表和验收条件，用于交叉对照 PRD 覆盖度 |

## 验收标准

### 文件存在性

- [ ] `{workspace}/prd-analysis.md` 文件存在且可读
- [ ] 文件非空（内容长度 > 0）
- [ ] 文件结构正确（包含 5-zone 章节结构）

### 完整性

- [ ] 5 个 zone 全部填充（Feature / User Story / Data Model / Constraint / Acceptance Criteria）
- [ ] 每个 Feature 有唯一编号（F1, F2...）和优先级
- [ ] 每个 Feature 有关联功能的引用（如有）

### 准确性

- [ ] PRD 的每个章节至少映射到一个 zone 的条目（覆盖率检查）
- [ ] 用户故事包含角色、操作和收益三要素
- [ ] 数据模型的实体之间关系已标注
- [ ] 约束条件已分类（性能/安全/兼容性/业务/合规）

### 歧义处理

- [ ] 检测到的歧义已列出并标注类型（模糊量词 / 缺失错误处理 / 未定义术语 / 矛盾需求 / 缺失边界）
- [ ] 每个歧义以具体问题形式呈现（非模糊担忧）
- [ ] 歧义已通过检查点交由用户确认

### 用户确认

- [ ] 用户确认分析结果符合 PRD 原意（检查点通过）
